# 9.2.0 Release-Candidate Notes

These notes describe the intended `9.2.0` candidate. They do not claim that a tag was created, the plugin was published, or live fresh-install smoke tests passed.

## Summary

- Adds capability preflight and requested-versus-actual route evidence with explicit degraded modes for Codex and Claude Code.
- Adds model-agnostic adaptive reasoning: every bounded task is classified `low|medium|high`, then each host adapter translates the requested tier through fresh runtime capabilities without static model-slug tables.
- Keeps the orchestrator state/event contract at `1.0.0`; runtime, routing, and approval display data remain additive metadata or linked artifacts.
- Makes immutable schema-valid `decision.recorded` evidence the compatibility authorization source and fails closed on metadata-only intent.
- Makes `ant:merge-request` the sole PR/MR creation/update owner while retaining the delivery-workflows legacy entrypoint as a deprecated forwarding bridge.

## Compatibility Evidence

| Scenario | Expected `9.2.0` result | Evidence surface |
|---|---|---|
| Runtime supports nested delegation | Use the observed nested route | `metadata.runtime`, `metadata.routingDecisions`, linked preflight artifact |
| Runtime cannot confirm required depth | Use a flat root-dispatch route | route fallback/degraded-mode evidence |
| Requested model/reasoning differs from host result | Preserve requested and actual separately | host-returned evidence or `actual: unknown` |
| Reasoning selector is supported but application is not reported | Preserve requested canonical/host values and translation evidence; keep actual host value and tier `unknown` | route evidence plus supported-unobservable evals for both hosts |
| Reasoning capability is unsupported, unknown, or stale | Omit speculative selector, use the documented host/session default, and record a specific fallback reason | capability artifact and route decision |
| Permission-sensitive work needs foreground retry | Change execution mode without changing or claiming the reasoning tier | independent execution and reasoning evidence |
| Delivery metadata exists without authoritative approval | Deny edit/delivery and request approval | resolver result plus contract examples |
| Grant is superseded or revoked | Deny the stale/revoked approval | replayed `decision.recorded` events |
| Orchestrator hands off PR/MR evidence | Canonical merge-request skill owns the complete provider/language/content/confirmation/command flow | handoff manifest plus duplicate-logic audit |
| `9.2.0` is rolled back to `9.1.0` | Existing state `1.0.0` remains readable; additive metadata may be ignored | schema validation and rollback smoke |

## Required Release Checks

- validate every state and JSONL example against the contract `1.0.0` schemas;
- validate all JSON manifests and exact equality of the three `9.2.0` version authorities;
- run `claude plugin validate .` and `claude plugin validate ./plugins/ant`;
- run the orchestrator synthetic specification suite and review every must-change diff; do not treat checked-in fixtures as live-host evidence;
- require a paired provenance manifest for external traces and claim live-host evidence only for validated `live-host-capture` provenance;
- verify adaptive-reasoning cases across all complexity tiers, both hosts, supported/unsupported/unknown and supported-but-unobservable capability states, resume revalidation, and permission orthogonality;
- run fresh Codex install/restart/invocation and `9.2.0 -> 9.1.0` rollback smoke;
- run fresh Claude Code marketplace install/invocation plus update/reload and rollback smoke;
- reject the candidate for any open P0; accept P1/P2 only with a valid finding-specific typed risk acceptance.

Fresh-install, network, update/reload, rollback, and publish evidence must come from real host runs. Local manifest/schema validation is not a substitute. Publishing, tagging, or releasing requires separate explicit approval.

## Rollback

1. Stop automatic delivery and preserve the run artifacts.
2. Restore the exact previously verified `9.1.0` tag/revision through the same Codex or Claude Code installation channel used for the candidate.
3. Restart Codex or update/reload Claude Code, then invoke the orchestrator in a new session.
4. Reopen the existing run and verify that the `1.0.0` snapshot/events remain readable; `9.1.0` may ignore additive `9.2.0` metadata.
5. Treat approvals as unknown unless the active runtime can verify the immutable event/artifact and digest. Request fresh approval before any mutation or delivery.
6. Record the failed scenario and block release until it is corrected and revalidated.
