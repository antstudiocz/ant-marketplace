orchestration logic

One root. One responsibility.

The orchestrator is not a visual map of everything at once. It is a chain of clear decisions that moves work from request to verified outcome.

root

Root directs. It does not write.

Root keeps user context, routing, progress, and delivery intent. Source code and changes belong to delegated roles.

Communication asks for decisions that cannot be safely inferred from the repository.
Scale chooses the smallest agent shape that can complete the work safely.
Evidence does not accept a worker report as done without verification or review.
lifecycle

Root sends the token.

Root scopes the work, scouts gather facts when needed, the lead implements, and a reviewer checks risky results.

routing

Complexity chooses the shape.

Speed does not come from bypassing rules. It comes from choosing the smallest delegated setup that is still safe for the work.

Simple one implementation owner and focused checks.
Standard one lead; scout or reviewer only when useful.
High assurance scouts as needed, one lead, and independent review.
No bookkeeping no phase owners, per-file agents, or custom runtime.
before code

Facts first. Plan second.

A scout separates repository truth from user decisions. Root or the lead turns that into the smallest plan that removes implementation ambiguity.

Scout report current behavior, contracts, obsolete paths, options, and blocking questions.
Direction recommendation with tradeoffs, not blind acceptance of the first idea.
Plan acceptance scenarios, rollout, validation, and decision policy.
Reads code checks the plan against the real repository state, not against an assumption.
Splits work creates bounded slices only where a clear contract exists.
Integrates combines the result, runs targeted checks, and prepares the evidence trail.
delegated implementation

Lead owns the change.

Even a small follow-up stays delegated while orchestration is active. Root-cause fixes beat workarounds and suppressed errors.

review / fix

Done is not a worker claim.

Reviewer does not start with the diff alone. It reads the goal, relevant context, reports, checks, and risks that can block completion.

Finding scope mismatch, contract break, missing tests, or weak evidence.
Fix lead fixes the root cause and adds targeted verification.
Re-review a blocking finding closes only after fix, verification, and focused re-review.
adaptive execution

The workflow changes with the work.

Native host controls keep the process responsive without a custom runtime. New complexity, user input, and validation results change only what they affect.

Adaptive reasoning escalates for uncertainty and de-escalates for deterministic segments.
Live user input updates impacted work without pausing independent progress.
Targeted checks run after coherent implementation phases, not after every edit.
One final suite runs on the exact final tree before delivery.
